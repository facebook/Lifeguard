# Effects and Errors

This document explains the two core concepts in Lifeguard's analysis pipeline: **effects** (observed side effects in Python code) and **errors** (Lazy Imports incompatibilities derived from those effects). Effects are the raw observations; errors are the safety judgments.

## Effects

An effect is a single observed side effect at a specific location in Python source code. Effects are detected during AST traversal and recorded per-scope (module, class, or function).

### Core Types (`effects.rs`)

```rust
pub struct Effect {
    pub kind: EffectKind,   // What kind of side effect
    pub name: ModuleName,   // Fully qualified name of the entity involved
    pub range: TextRange,   // Source location (byte range)
    pub data: EffectData,   // Optional metadata (e.g., whether call args are unsafe)
}
```

`EffectData` is either `None` or `Call(CallData { has_unsafe_args })`, indicating whether any argument to a call is an imported variable.

### EffectKind

`EffectKind` is an enum. See the variant comments in `effects.rs` for descriptions of each variant.

### Key EffectKind Methods

- **`is_runnable()`** — Returns true for call effects where the analyzer can recurse into the called function body: `FunctionCall`, `ImportedFunctionCall`, `DecoratorCall`, `ImportedDecoratorCall`, `MethodCall`. These trigger call graph traversal in `project.rs`.

- **`requires_eager_loading_imports()`** — Returns true for `CustomFinalizer`, `ExecCall`, `SysModulesAccess`. These cause the module to be added to the `load_imports_eagerly` set regardless of scope.

- **`is_unsafe_stub_effect()`** — Returns true for `UnknownEffects`, `Unsafe`, `Mutation`. Used for stub file analysis.

### EffectTable

`EffectTable` is a `HashMap<ModuleName, Vec<Effect>>` that maps scope names (fully qualified module/class/function names) to the effects observed in that scope. This scoping is critical: effects in a function body are recorded under the function's qualified name, not the module's.

### Effect Generation

Effects are generated by two analyzers depending on file type:

**Source files (`.py`) — `source_analyzer.rs`:**
The `SourceAnalyzer` walks the Python AST and produces effects for each statement type:
- Assignments check for imported variable or global targets
- Function/method calls are resolved and classified (imported, local, unknown)
- Decorators are treated as function calls
- Function/class bodies are analyzed in their own scope
- `raise` statements produce effects unless inside a `try:` block
- Special builtins (`exec`, `getattr`/`setattr` with non-literal args, `sys.modules`) get dedicated effect kinds

**Stub files (`.pyi`) — `stub_analyzer.rs`:**
Stubs declare effects explicitly using pseudo-function calls in the body:
```python
def some_function():
    imported_function_call("os.path.join")  # declares this effect
    mutation()                               # declares this is mutating
    no_effects()                            # declares this is safe
```
Functions with body `...` and no annotations automatically get `UnknownEffects`.

## Errors

An error (formally `SafetyError`) represents a Lazy Imports incompatibility — a reason why a module cannot be safely lazily imported.

### Core Types (`errors.rs`)

```rust
pub struct SafetyError {
    pub kind: ErrorKind,
    pub metadata: ErrorMetadata,  // interned string (usually the entity name)
    pub range: TextRange,
}
```

### ErrorKind

`ErrorKind` is a variant enum:

| ErrorKind | Meaning |
|-----------|---------|
| `UnsafeDecoratorCall` | Decorator with side effects |
| `UnsafeFunctionCall` | Function call with side effects |
| `UnsafeMethodCall` | Method call with side effects |
| `ProhibitedCall` | Legacy alias for `UnsafeFunctionCall` |
| `UnknownDecoratorCall` | Unresolved decorator |
| `UnknownFunctionCall` | Unresolved function call |
| `UnknownMethodCall` | Unresolved method call |
| `UnknownObject` | Unresolved object access |
| `UnhandledException` | Unhandled `raise` at module level |
| `CustomFinalizer` | Class with `__del__` |
| `ExecCall` | `exec()` call |
| `SysModulesAccess` | `sys.modules` access |
| `SubmoduleReExport` | Submodule re-export name collision |
| `ImportedModuleAssignment` | Assigning to an imported module's attribute |
| `ImportedVarArgument` | Passing imported var to a function that mutates params |
| `UnknownEffects` | Stub declares unknown effects |

### ModuleSafety (`module_safety.rs`)

`ModuleSafety` holds the final safety verdict for a module:

```rust
pub struct ModuleSafety {
    pub errors: Vec<SafetyError>,
    pub force_imports_eager_overrides: Vec<SafetyError>,
    pub implicit_imports: Vec<ModuleName>,
}
```

- A module `is_safe()` when `errors` is empty.
- A module `should_load_imports_eagerly()` when `force_imports_eager_overrides` is non-empty (only `CustomFinalizer`, `ExecCall`, `SysModulesAccess`).

`SafetyResult` wraps either `Ok(ModuleSafety)` or `AnalysisError(anyhow::Error)`.

## The Effect-to-Error Conversion Pipeline

The conversion from effects to errors happens in `project.rs` and is the heart of Lifeguard's analysis.

### Step 1: Merge Effects

All per-module `EffectTable`s are merged into a single global table (`merge_all_effects`). Nested scope effects are propagated upward — a decorator wrapping a function with side effects is itself marked as having side effects.

### Step 2: Build ProjectInfo

`ProjectInfo` aggregates the global effect table, class table, function-to-module mappings, and re-export information.

### Step 3: Collect Errors

`collect_errors_from_project` iterates over each module's **eager scopes** (module-level and class-level code that executes at import time) and converts effects to errors through three mechanisms:

1. **Direct conversion** (`SafetyError::from_effect`): Some effects unconditionally become errors:
   - `ProhibitedFunctionCall` → `ProhibitedCall`
   - `UnknownFunctionCall` → `UnknownFunctionCall`
   - `Raise` → `UnhandledException`
   - `CustomFinalizer` → `CustomFinalizer`
   - `ExecCall` → `ExecCall`
   - `SysModulesAccess` → `SysModulesAccess`
   - `UnknownDecoratorCall` → `UnknownDecoratorCall`
   - `UnknownEffects` → `UnknownEffects`
   - `UnknownObject` → `UnknownObject`

2. **Call graph traversal** (`SafetyError::from_unsafe_call`): For "runnable" effects (`FunctionCall`, `MethodCall`, `DecoratorCall`, etc.), the analyzer recursively walks into the called function's body to determine safety. This uses:
   - A `CallStack` to prevent infinite recursion
   - A `FunctionSafety` cache with three states: `Safe`, `Unsafe`, and `UnsafeIfImported` (safe when called within the same module, unsafe cross-module due to global variable mutation)

3. **Special handling**:
   - `ImportedVarMutation` at module scope → `ImportedModuleAssignment` error
   - `ImportedTypeAttr` → checked for property access, treated as a call
   - `ImportedVarArgument` + `ParamMethodCall` → `ImportedVarArgument` error
   - Effects with `requires_eager_loading_imports()` → added to `force_imports_eager_overrides`
   - `SubmoduleReExport` effects → attached to the submodule being re-exported

### Not All Effects Become Errors

Many effect kinds exist only for call graph traversal or state tracking:

- `FunctionCall`, `ImportedFunctionCall`, `MethodCall`, `DecoratorCall`, `ImportedDecoratorCall` — only become errors if the called function is determined unsafe after body analysis
- `GlobalVarAssign`, `GlobalVarMutation` — only become errors in cross-module calls (`UnsafeIfImported`)
- `ParamMethodCall` — only becomes an error if the call has `has_unsafe_args: true`
- `ClassVarAssign`, `SetAttr`, `SetSubscript` — tracked as effects but not directly converted to errors
- `NoEffects`, `Mutation`, `Dunder` — stub-only markers, never directly become errors

## Pipeline Summary

```
Python AST
    │
    ▼  [source_analyzer.rs / stub_analyzer.rs]
Effects (EffectTable: scope → Vec<Effect>)
    │
    ▼  [project.rs: collect_errors_from_project()]
    │
    ├── Direct effects (UnknownFunctionCall, Raise, etc.) ──→ SafetyError
    ├── Runnable effects (FunctionCall, etc.) ──→ Call graph traversal ──→ Safe or SafetyError
    ├── Eager-loading effects (CustomFinalizer, etc.) ──→ force_imports_eager_overrides
    └── Mutation effects (GlobalVarAssign) ──→ UnsafeIfImported (context-dependent)
    │
    ▼
ModuleSafety (errors + force_imports_eager_overrides + implicit_imports)
    │
    ▼  [output.rs: LifeGuardAnalysis::new()]
LifeGuardOutput (lazy_eligible dict + load_imports_eagerly set)
```

## Key Files

| File | Role |
|------|------|
| `effects.rs` | `Effect`, `EffectKind`, `EffectTable`, `CallKind`, `EffectData` |
| `errors.rs` | `SafetyError`, `ErrorKind`, effect-to-error conversion methods |
| `module_effects.rs` | `ModuleEffects` — per-module effect accumulator; also `FileError` |
| `module_safety.rs` | `ModuleSafety`, `SafetyResult` — per-module safety verdict |
| `source_analyzer.rs` | Generates effects from `.py` AST traversal |
| `stub_analyzer.rs` | Parses effect declarations from `.pyi` stub files |
| `project.rs` | Converts effects to errors via call graph analysis |
| `output.rs` | Converts safety map to final output |
| `cursor.rs` | Tracks scope during AST traversal (determines where effects are recorded) |
| `bin/show_effects.rs` | CLI tool to dump effects for a single file |
